## 2. Relevant User Flows

### Flow 4: Parent Receives and Acts on Push Notification (Design.md §4)
1. Notification arrives: `"New rhyme added — check it out! 🎵"`
2. Parent taps notification → app opens directly to Video Library (`/library`)
3. New video appears at top of grid (sorted by `publishedAt DESC`)
4. Parent taps new video → plays

Success state: seamless open to `/library`; new video at top.
Error state: app not installed / notification dismissed → no action needed.

### Opt-in sub-flow (Story 9 specific):
1. First video ends → `NotificationOptInSheet` slides up
2. "Yes, notify me" → browser `Notification.requestPermission()` → FCM `getToken()` → Firestore write (`fcmToken`, `notificationsEnabled: true`)
3. "Not now" → sheet dismisses → no Firestore change; localStorage flag `ckd_notif_prompted` set to prevent re-prompt
4. Settings toggle → Parental Gate → if passed → toggle `notificationsEnabled` in Firestore

## 6. Web FCM Requirements (Story-specific)

- **Service worker**: `apps/web/public/firebase-messaging-sw.js` — required for background message handling and notification click → open `/library`
- **VAPID key**: stored in `.env.local` as `VITE_VAPID_KEY` — passed to `getToken()` call
- **Permission flow**: `Notification.requestPermission()` (browser) → on 'granted' → `getToken(messaging, { vapidKey, serviceWorkerRegistration })`
- **Token refresh**: call `getToken()` on each app load; compare with stored token; update Firestore if different
- **No `onTokenRefresh` listener on web**: web FCM does not expose a token-rotation listener — periodic re-check on mount handles this
